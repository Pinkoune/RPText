Simulation tour-par-tour · toolkit balance-sim-turns

Équilibrage des 20 classes au Niveau 50

Chaque classe maxée (talents + gear cible) affronte un boss calibré. Deux mesures : l'ATK effective (proxy de DPS, axe horizontal) et la Survie — PV restants en fin de combat (axe vertical). La bande bleue est la zone saine visée. Toutes les sous-classes y logent ; seules les bases Mage/Archer tombent dessous — attendu, on ascensionne dès le Nv.20.

★ Nouvelles classes de ce patch — Piégeur (Archer), Oracle (Soigneur), Nécromancien (Mage, + invocation « Lever un mort »). Toutes atterrissent dans la bande. ◇ bases non-ascensionnées. Lecture : haut = increvable (tanks/soigneurs), droite = gros dégâts (Pyromancien 877 ATK, Berserker/Nécromancien ~720).

Reprendre la tâche — pour l'agent suivant

Le système de classes est entièrement data-driven : une sous-classe apparaît automatiquement dans l'écran d'ascension (TalentCard) et le Wiki via getAscensions(base.id) / CLASS_LIST — aucun câblage UI. Pour en ajouter une :

  1. Type — ajoute l'id à l'union ClassId dans src/game/types.ts (dans le groupe de sa famille).
  2. Définition — entrée dans CLASSES (src/game/classes.ts) : parent (classe de base), base, growth, emoji, desc, playstyle (texte Wiki).
  3. Arbre de talents — src/game/talents.ts : ~21 rangs propres (budget Nv.50 = base 27 + sous-classe 21 = 48). Gabarit : 1 skill d'ouverture + branches passives + 1 finisher gated par ressource + régén. Ajoute aussi l'inné dans talentMods (ou hérite de la famille), et l'entrée dans classResourceType + RESOURCE_INFO.
  4. Ressource (si exotique) — le finisher porte resource:{type,cost}. Ajoute le nom au type union de ActiveSkillDef.resource ET du param resourceType de combatTurn, puis le déclencheur de gain dans combat.ts (bloc resourceGained, sert chasse + aventure + endless-solo) et son miroir dans endlessService.ts (2 blocs : tour monstre / tour joueur, endless-multi). Enfin RESOURCE_META dans HuntCard.tsx (label + couleur de barre).
  5. Valide — lance la simulation tour-par-tour (voir ci-dessous). Elle teste chaque classe de CLASS_LIST sans code sup → toute classe future est couverte d'office.

Simulation (bundlée esbuild, Firebase stubé, exécutée node) :

node_modules/.bin/esbuild scripts/balance-sim-turns.ts --bundle --platform=node \
  --format=cjs --define:import.meta.env='{}' --outfile=/tmp/bt.cjs \
  && node /tmp/bt.cjs

Bandes cibles Nv.50 (tour-par-tour, classe maxée) : Winrate 100%, Survie (endHP) 35–100%. En dessous de 35% = trop fragile ; les bases Mage/Archer y échappent volontairement (on ascensionne au Nv.20). Détails dans scripts/README-balance.md.
